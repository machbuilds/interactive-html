Server-Sent Events: the underrated push channel

WebSockets get the headlines, but for the most common real-time need — a server telling a page "something changed" — there's a simpler tool that's been sitting in browsers for over a decade.

What SSE actually is

Server-Sent Events is a one-way push channel over plain HTTP. The browser opens a single long-lived request with EventSource, and the server keeps the response open, writing small text frames whenever it has news. No handshake upgrade, no binary framing, no new protocol — it's just an HTTP response that never finishes.

That simplicity is the point. If your page only needs to receive updates — notifications, progress, live scores, "the file changed, refetch it" — SSE does the job with a fraction of the moving parts a WebSocket needs.

The lifecycle, end to end

That auto-reconnect is SSE's killer feature. With WebSockets, reconnection logic — backoff, resubscription, missed-message recovery — is your job, and everyone writes it slightly wrong. EventSource ships it in the platform: the browser retries on its own and sends Last-Event-ID so the server can resume the stream without gaps.

The wire format is just text

Frames are UTF-8 lines separated by a blank line. A minimal stream looks like this:

HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache

event: score
id: 7
data: {"home": 2, "away": 1}

: this line is a comment — handy as a keep-alive ping

data: a frame with no event name fires "message"

Four field names total — event, data, id, retry — plus comment lines starting with :. You can implement a conforming server in any language in a screenful of code; curl is a perfectly good debugging client.

Rule of thumb: if the client never needs to push messages up the same channel, you probably don't need a WebSocket. Send client→server traffic as ordinary fetch requests and let SSE handle the downstream. You keep HTTP semantics — auth headers, status codes, caching rules, load balancers — for free.

The gotchas worth knowing

Proxy buffering

Reverse proxies love to buffer responses, which turns your real-time stream into a batch job. Nginx honors the X-Accel-Buffering: no response header; for others, disable buffering for the stream's route. Sending a comment line as a periodic ping also defeats idle timeouts along the path.

The six-connection ceiling (HTTP/1.1 only)

Browsers cap HTTP/1.1 at roughly six connections per origin. Six tabs each holding an SSE stream will starve the seventh. Over HTTP/2 the streams multiplex onto one connection and the ceiling effectively disappears — serve over HTTP/2 if you expect many tabs.

Text only

Payloads are UTF-8 text. In practice everyone sends JSON and never notices; if you genuinely need binary push, that's a real reason to reach for WebSockets.

When to choose what

SSE quietly powers more than people think: ChatGPT-style token streaming, build-progress UIs, log tails, stock tickers, and the live-reload loop in the page you're reading this on. It's not the flashy choice. It's the one you don't have to debug at 2am.
